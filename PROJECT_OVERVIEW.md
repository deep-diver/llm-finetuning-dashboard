# Project Overview: Fine-Tuning Tutorial Architecture

This project is structured as a modular, contract-driven system that separates data storage, training configurations, scripting interfaces, core package logic, and agent instructions.

## Directory Roles

* **`.agents/skills/`**: Houses the instruction manuals (`SKILL.md`) for each agent type. When AntiGravity performs a stage of the pipeline, it references these files to understand its responsibilities, input constraints, output contracts, and handoff conditions.
* **`configs/`**: Separates base defaults from active experiments:
  * `base/`: Baseline profiles specifying model, dataset, training, hardware, and evaluation defaults.
  * `experiments/`: Dynamic, run-specific configurations generated by the Orchestrator or training-configuration agent.
* **`schemas/`**: Declares JSON schemas that define structured interfaces between agents. Every JSON report or manifest generated by an agent must be validated against its respective schema in this directory.
* **`data/`**: Structured data storage:
  * `raw/`: Unprocessed, downloaded source datasets (immutable).
  * `processed/`: Cleaned, tokenized, and split files (`train.jsonl`, `valid.jsonl`, `test.jsonl`).
  * `synthetic/`: Generated augmentations targeting specific model failure modes.
  * `cache/`: Intermediate files, tokenizers, or downloaded model checkpoints.
* **`scripts/`**: Executables for running each stage. They act as light CLI entry points wrapping python libraries under `src/`.
* **`src/finetuning_lifecycle/`**: The core Python package containing logic for config loading, artifact validation, data quality checks, metrics calculation, and reporter formatters.
* **`runs/`**: Target directory for run-specific artifacts. Output structure: `runs/<run_id>/`.
* **`reports/`**: Final, human-readable summary documents outlining experiment results and recommendations.
* **`tests/`**: Unit tests that validate script arguments, schema checks, and pipeline sanity.

## Core Design Principles

1. **Explicit Artifact Contracts**:
   Instead of using implicit states, agents communicate by producing files that match specific JSON schemas. If `dataset-preparation` produces an invalid `dataset_report.json`, the orchestrator halts execution before config generation.
   
2. **Auto-Research via Hyperparameter Loop**:
   Before running a multi-hour, multi-node fine-tuning job on expensive TPU instances, the system executes rapid few-step local runs. The local evaluation agent returns metrics which AntiGravity reviews, proposing parameter updates to find a stable configuration.
   
3. **Safety and Cost Control**:
   - Small models and tiny datasets are used by default.
   - All gcloud commands are validated prior to execution.
   - Large or full training runs require explicit user authorization.
   - Run paths (`runs/<run_id>/`) are isolated to prevent overwriting historical experiments.
